> **Статус:** ПОДТВЕРЖДЁННЫЙ БАГ, требует фикса. Зафиксировано 2026-06-16

## TL;DR (суть за 30 секунд)

Claude Code пишет **один** ответ модели (один API-вызов = один `message.id`)
**несколькими строками** `.jsonl` — по одной строке на каждый блок контента
(`thinking`, `text`, и каждый `tool_use`). И в **каждую** из этих строк
кладётся **один и тот же** блок `usage`.

`sumTranscript()` в [`src/metrics.ts`](../src/metrics.ts) суммирует `usage`
**по строкам** → один API-вызов считается 2–4 раза. На реальных сессиях
владельца **абсолютные** числа (work / cache_read / cache_write / effective /
«стоимость») завышены в **~2.3–3.3 раза**.

**Что искажено:** все абсолютные токен-числа и cost/`effective`/`N× cheaper`.
**Что НЕ искажено:** cache **hit-rate** (% — почти не меняется, см. ниже),
context-window % и квоты 5h/7d (они берутся не из суммы транскрипта).

**Фикс:** дедупликация по `message.id` — считать `usage` каждого уникального
ответа модели **ровно один раз**.

---

## 1. Симптом, с которого начали (наблюдение владельца)

> «Утром, холодный старт, кэша ещё быть не должно. Но как только запускаю агента
> (`продолжаем`), он только начинает грузить файлы в кэш — а расширение уже
> показывает чтение из кэша. Подозрение: кэш считается неправильно.»

Подозрение **верное** — счёт неправильный, но не там, где казалось. Разберём
ниже два независимых эффекта: (1) реальный баг дублирования, (2) реальная
механика «запись-затем-чтение» внутри одного хода.

---

## 2. Корневая причина

Формат транскрипта Claude Code (текущие версии, Opus 4.8/1M): ответ ассистента
с несколькими блоками контента сериализуется как **несколько** строк `.jsonl`.
Все строки одного ответа имеют:

- **одинаковый** `message.id` (напр. `msg_01QCBVi9rR8TRi`)
- **одинаковый** `requestId` (напр. `req_011Cc6UDgFs9ch`)
- **одинаковый** `message.usage` (повторяется дословно)
- **разные** `uuid` (каждая строка — отдельная запись) и разные блоки `content`

### Реальный дамп (первый ответ текущей сессии, холодный старт)

```
line#1: uuid=d1f8f229  msg.id=msg_01QCBVi9rR8TRi  reqId=req_011Cc6UDgFs9ch  blocks=[thinking]   out=1092 cache_read=13611
line#2: uuid=17e9fdf6  msg.id=msg_01QCBVi9rR8TRi  reqId=req_011Cc6UDgFs9ch  blocks=[text]       out=1092 cache_read=13611
line#3: uuid=aec4192e  msg.id=msg_01QCBVi9rR8TRi  reqId=req_011Cc6UDgFs9ch  blocks=[tool_use]   out=1092 cache_read=13611
line#4: uuid=58b4ffea  msg.id=msg_01QCBVi9rR8TRi  reqId=req_011Cc6UDgFs9ch  blocks=[tool_use]   out=1092 cache_read=13611
line#5: uuid=bd9923d1  msg.id=msg_01MMJnQMBm1Y8U  reqId=req_011Cc6UHPkJCWJ  blocks=[thinking]   out=468  cache_read=47745
```

`output_tokens=1092` **дословно совпадает** на строках 1–4. Четыре разные
генерации модели не могут дать побайтово одинаковый счёт выходных токенов при
одном `message.id`. Значит это **один** ответ, записанный 4 раза. `sumTranscript`
прибавляет его `usage` 4 раза.

## 3. Доказательства масштаба (6 последних реальных сессий)

Считал двумя способами: **RAW** = текущая логика (сумма по строкам) против
**DEDUP** = сумма по уникальному `message.id`.

| транскрипт   | строк | уник. msg | строк/msg | work ×завыш | effective ×завыш | cache_read 1-го вызова |
|--------------|------:|----------:|----------:|------------:|-----------------:|-----------------------:|
| 97afeb4f…    |    93 |        32 |     2.91  |     3.30    |      2.94        | 13 611 |
| 5ce1603b…    |   365 |       154 |     2.37  |     2.73    |      2.34        | 13 448 |
| 170afd56…    |   339 |       149 |     2.28  |     2.85    |      2.36        | 13 448 |
| 9a5aa733…    |   266 |       116 |     2.29  |     2.69    |      2.30        | 13 448 |
| 3f4cd8d8…    |   257 |       115 |     2.23  |     2.72    |      2.25        | 13 448 |
| acfe5f13…    |   358 |       160 |     2.24  |     2.71    |      2.28        | 13 448 |

Коэффициент завышения **не константа** — он зависит от среднего числа блоков на
ответ (ответ из одного `text` = 1 строка, ответ `thinking+text+3×tool_use` = 5
строк). Поэтому **нельзя** «делить на 3»; нужен честный dedup.

## 4. Что именно искажено, а что — нет

| Метрика | Источник | Искажена багом? |
|---|---|---|
| work (input+output), absolute | сумма транскрипта | **ДА, ~2.7–3.3×** |
| cache_read / cache_write, absolute | сумма транскрипта | **ДА, ~2.3–3×** |
| `effective` (cache-weighted) + «N× cheaper» / cost-блок | сумма транскрипта | **ДА, ~2.3–2.9×** |
| **cache hit-rate (%)** | `cacheRead/(cacheRead+cacheWrite+input)` | **Практически НЕТ** — числитель и знаменатель раздуваются вместе; замер дрейфа: ≤0.73 п.п. на 3 сессиях |
| **context-window %** | `lastAssistantContext()` — последний ход, НЕ сумма | **НЕТ** — берёт usage последней строки, дубли идемпотентны |
| **cache tier (1h/5m)** | `lastCacheTier()` — последний write-ход | **НЕТ** — та же причина |
| **квоты 5h / 7d** | `parseRateLimitHeaders()` из HTTP-заголовков API | **НЕТ** — не зависит от транскрипта |

Вывод: коллапс-бар (квоты + context%) и hit-rate — корректны. **Завышены
абсолютные токен-числа и весь cost/`effective`-блок в tooltip/панели.**

## 5. Ответ на исходный вопрос про «утренний кэш»

`cache_read > 0` на первом же ходу холодного старта — это **сумма трёх**
эффектов, два из которых реальны:

1. **Артефакт подсчёта (главный визуальный вклад):** первый ответ записан ~2–4
   строками с повтором `usage` → видимый `cache_read` сразу умножается. Лечится
   dedup'ом.

2. **Реальная механика агент-цикла (интуиция владельца — верна):** одно сообщение
   `продолжаем` порождает **много последовательных** API-вызовов (по одному на
   round-trip инструмента). Первый вызов **пишет** контекст в кэш
   (`cache_creation`), каждый следующий в том же ходу **читает** его обратно
   (`cache_read`). В дампе: вызов №2 (`req…UHP`, строка 5) уже читает 47 745 —
   это ровно то, что вызов №1 записал. Т.е. «быстро загнал в кэш и тут же
   читаю» — буквально так и есть.

3. **Стабильный warm-сегмент системного промпта:** `cache_read` **самого первого**
   вызова ≈ **13.4–13.6 тыс. в КАЖДОЙ сессии** (см. последний столбец таблицы).
   Такая стабильность означает: это один и тот же большой системный
   промпт + определения инструментов Claude Code (идентичны во всех сессиях и
   проектах). Anthropic кэширует по хэшу контента на уровне аккаунта, и этот
   сегмент остаётся «тёплым» между запусками (его греет любая недавняя
   активность Claude в любом проекте). Поэтому проект «холодный», а универсальный
   системный сегмент — нет.

Итого: настоящего «протёкшего вчерашнего кэша этого проекта» здесь нет. Есть
×2–3 артефакт + честный self-read агент-цикла + общий тёплый системный промпт.

## 6. Как чинить (правильно)

### Принцип
Anthropic API возвращает `usage` **один раз на ответ** (на один `message.id` /
`requestId`). Значит `usage` нужно учитывать **один раз на уникальный
`message.id`**, а не на строку.

### Патч `sumTranscript()` в `src/metrics.ts`

```ts
export function sumTranscript(raw: string): Totals {
  const t = emptyTotals();
  const seen = new Set<string>();              // dedup by API response id
  for (const line of raw.split(/\r?\n/)) {
    const s = line.trim();
    if (!s) continue;
    let obj: any;
    try { obj = JSON.parse(s); } catch { continue; }
    if (obj?.type !== "assistant" || !obj.message) continue;
    if (obj.isSidechain) continue;

    // One API response is split across multiple jsonl lines (thinking / text /
    // tool_use …), each repeating the SAME usage block. Count it once.
    const id = obj.message.id || obj.requestId;
    if (id) {
      if (seen.has(id)) continue;
      seen.add(id);
    }
    // (no id → very old/odd line → fall through and count it, as before)

    const u = obj.message.usage || {};
    t.input     += u.input_tokens || 0;
    t.output    += u.output_tokens || 0;
    t.cacheWrite += cacheWriteTokens(u);
    t.cacheRead += u.cache_read_input_tokens || 0;
  }
  t.work = t.input + t.output;
  return t;
}
```

### Что НЕ трогать
- `lastAssistantContext()` и `lastCacheTier()` — берут «последний» ход, дубли
  идемпотентны, фикс не нужен (и не навредит, но не обязателен).
- `cacheHitRatePct()` — формула верна; после фикса `sumTranscript` она просто
  станет считаться от дедуп-сумм (дрейф ≤1 п.п., незаметно).
- Веса `effective` (`cacheRead 0.1`, `cacheWrite 1.25`) — корректны, не менять.

### Тесты (юнит, в духе существующего `src/test/logic.test.ts`)
1. Три строки с одинаковым `message.id` и одинаковым `usage` → `sumTranscript`
   учитывает их **один раз** (work/cache = значения одного ответа, не ×3).
2. Строки с разными `message.id` → суммируются все (без потери).
3. Fallback: строки без `message.id`, но с `requestId` → dedup по `requestId`.
4. `isSidechain`-строки по-прежнему исключаются.
5. Регрессия: hit-rate, context%, tier — не изменились на фикстуре с дублями.

### Эталонная проверка на реальных данных
После фикса `sumTranscript(файл 97afeb4f)` должен дать work ≈ 61 тыс.
(не ≈205 тыс.), effective ≈ 553 тыс. (не ≈1.68 млн).

## 7. ВАЖНО: тот же баг есть в `session-cost.py` методологии Dashboard

[`src/metrics.ts`](../src/metrics.ts) (комментарий вверху файла) обязан быть
синхронен с `tools/session-cost.py` проекта Dashboard. Там **та же ошибка**:
функция `parse_session()` добавляет запись на каждую строку без dedup по
`message.id`. Это значит, что весь журнал `docs/session-costs.jsonl`
(≈165 сессий) и метрики `cost-metrics.md` / `cost-trend.py` исторически
**завышены в ~2.3–3.3×** (forward-only — задним числом не пересчитывалось).

Фикс должен быть зеркальным в обоих местах. **Эта часть (Dashboard
session-cost.py) чинится в чате Dashboard-методологии, не здесь** — но команде
cc-statusbar полезно знать, что «эталон синхронизации» сам содержит баг, и не
копировать его обратно.

## 8. Оговорки / что перепроверить в вашем чате

- Поведение «один ответ = N строк» наблюдалось на текущих версиях Claude Code
  (Opus 4.8). На очень старых транскриптах разбивки могло не быть (тогда
  строка/msg = 1 и dedup ничего не меняет — безопасно).
- `message.id` присутствует на всех assistant-строках в наблюдённых данных;
  `requestId` — запасной ключ. Если найдёте assistant-строки без обоих — решите,
  считать ли их (рекомендация: считать, чтобы не терять данные).
- Codex-ветку проверить отдельно — там другой источник (`codexAppServer` /
  локальная история), эта находка к ней напрямую не относится.
- Все цифры выше получены на транскриптах владельца в
  `~/.claude/projects/c--Users-Honor-Desktop-My-Projects-Dashbaord/`;
  воспроизводимы тем же dedup-by-`message.id` скриптом.
